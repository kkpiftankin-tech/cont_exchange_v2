# Роли и персоны

В системе различаются **четыре основных типа пользователей**. Каждый имеет своя мотивацию, информационный профиль, ожидания по SLA и риск-профиль.

## 1. Liquidity Trader (ликвидный трейдер)

- **Мотивация:** реализовать внешний запрос — ребалансировка, потоки капитала, снижение риска. Не пытается заработать на ценовом сигнале.
- **Тип информации:** нет приватной (inside); только внешние бизнес-факторы (мандат фонда, риск-лимиты, cash flow).
- **Действия:** отправляет крупные flow-orders / CSLO («купить/продать X за окно T») с выбранным профилем скорости (TWAP / VWAP / оптимальный).
- **Цель по стоимости:** минимизировать Implementation Shortfall, slippage к benchmarks (arrival, TWAP/VWAP), при заданном VAR/CVaR.
- **Риски:** ценовое движение во время растянутой сделки; VAR/CVaR портфеля; ошибка ввода/слишком позднего входа.
- **Инструменты системы:** модели оптимального исполнения, прогноз IS, scenario analysis.
- **Метрика успеха:** низкий IS, попадание в доверительный интервал VWAP, нет нарушений лимитов.

## 2. Market Maker / Liquidity Provider (LP)

- **Мотивация:** заработать на спреде, impact и комиссиях за предоставление ликвидности.
- **Тип информации:** модели price impact, маркет-данные, иногда faster inflow info; в Kryptonâlike дизайне токсичная инфо нейтрализуется механизмом.
- **Действия:** публикует и обновляет residual demand/supply CSLO-кривые с ограничениями по скорости и объёму.
- **Цель:** стабильный положительный PnL после учёта комиссий, контролируемая дисперсия PnL, приемлемый inventory-расклад.
- **Риски:** inventory risk (большой long/short), adverse selection, скачки волатильности.
- **Инструменты:** модели «priceâimpact vs speed», inventory-модели, hedge-политики между площадками.
- **Метрика успеха:** устойчивый PnL, выполнение risk-лимитов, низкая доля adverse-selection сделок.

## 3. Client Integrator / Broker / Execution Provider

- **Мотивация:** упаковать механизм FOB в удобные алгоритмы (TWAP/VWAP/smart-routing) и продать клиентам как «лучшее исполнение».
- **Тип информации:** агрегация клиентских заявок и маркет-данных; не обязан иметь «alpha».
- **Действия:** превращает клиентские дискретные приказы в нормализованные FlowOrder, маршрутизирует на биржу, может хеджировать остатки на внешних CEX/DEX/AMM.
- **Цель:** SLA по исполнению (IS, VWAP, latency), конкурентные комиссии, минимум рекламаций.
- **Риски:** операционный и репутационный — плохое исполнение, breach лимитов, ошибки routing'а; рыночный риск ограничен (не держит открытые позиции).
- **Инструменты:** алгоритмы TWAP/VWAP поверх FOB/CSLO, smart-routing, backtest/replay, агрегированная аналитика IS/VWAP.

## 4. Operator / Administrator

- **Мотивация:** обеспечить надёжную, устойчивую, регуляторно-чистую работу рынка и монетизацию инфраструктуры.
- **Тип информации:** полный технический и операционный обзор: метрики, логи, риск-отчёты, соответствие KYC/AML.
- **Действия:** настройка matching-механизма, risk-движка, collateral-политики, комиссий, режимов работы (batchIntervalMs, лимиты по объёму, маржа).
- **Цель:** баланс доходов (комиссии, листинги, MM-программы) с качеством рынка (ликвидность, спреды, справедливость) и регуляторными требованиями.
- **Риски:** системный — сбои matching, ошибки risk-движка, неверные лимиты, утечка данных, регуляторные санкции.
- **Инструменты:** risk-engine (VAR/CVaR, лимиты), мониторинг качества рынка (спреды, глубина, fill-rate), kill-switch, KYC/AML проверки, аудит.

## Профили использования

Для каждой роли в системе должны быть определены:

- **Доступные типы операций** и ограничения (например, demo не может вывести средства, provider не может править solverconfig).
- **Права доступа к информации** (маркет-данные, отчётность, риск-индикаторы).
- **Ограничения по размеру, скорости, плечу.**

Это конфигурируется через таблицу `risk_limits` (см. [storage-schema.yaml](../../specs/domain/storage-schema.yaml)).
