## Conflicts

1. **Kafka topic split** (`batch.outputs` для BatchResult vs `fills` для FillEvent).
   - Текущее состояние repo: единый топик `batch.outputs` несёт оба типа сообщений.
   - Источник IN-003 предполагает раздельные топики `batch.outputs` и `fills`.
   - Резолюция: оставить single-topic как текущую реализацию MVP; добавить conflict note и пункт в task plan «future enhancement: split topics».
2. **Provider-Client Backend как отдельный контейнер.**
   - В IN-003 в техническом диаграмме виден `Provider-Client Backend` как отдельный сервис.
   - В нашей C2 этот слой = `order-flow`. Резолюция: в нормализованных артефактах используем имя `order-flow` (= provider-client), указываем эквивалентность в conflict note.
3. **`fills` table в OLTP.** В §4.4.3 сказано «`saveFills` в OLTP». В нашей схеме `fills` хранятся только в OLAP (ClickHouse), а OLTP содержит `flow_orders` со `filled_cum`. Резолюция: оставить текущую схему (OLAP-only); рассматривать `saveFills` как абстракцию «записать в долговременное хранилище через Kafka → ClickHouse».
