## Принцип

Документация — это не приложение к коду, а **рабочий механизм управления проектом**.
Требования, архитектура, домен, API, данные, инфраструктура, тесты и эксплуатация
живут в репозитории, версионируются и проверяются вместе с кодом.

## Правила

1. Документация в git рядом с кодом.
2. Новые материалы сначала кладутся в `incoming-docs/`.
3. После обработки раскладываются по `docs/0X-*/`.
4. Каждое архитектурное решение → ADR.
5. Каждый контракт фиксируется **до** реализации кода.
6. Каждая фича имеет трассировку: **business goal → requirement → use case → component → API → data → test → runbook**.
7. Код не вводит бизнес-семантику, отсутствующую в документации.
8. При расхождении кода и доков нужно обновить документацию, исправить код или зафиксировать ADR.

## Правила качества

1. Каждый документ имеет понятную цель.
2. Каждый документ ссылается на связанные.
3. Каждый термин — в глоссарии или Ubiquitous Language.
4. Диаграммы как текст (Mermaid, PlantUML, Structurizr DSL).
5. Никаких секретов в документах.
6. Документы пригодны для автоматической валидации (markdownlint, link checker, YAML schema).
7. TODO имеют owner и срок.
8. Финансовые величины описаны с единицами (base, quote, price, notional, fee, margin).
